# Library Module — Design Spec

## Overview

The Library module is Hlusra's central data service. It owns the SQLite database and filesystem structure, and is the single point of access for all meeting data. No other module reads or writes to the database or meeting directories directly — everything goes through the Library API.

## Architecture

The Library acts as a service layer between all other modules and the data:

- **Recorder** → `Library::prepare_meeting()` before recording starts (gets ID + dir path), then `Library::finalize_meeting()` after recording stops
- **Transcription** → `Library::save_artifact()` + `Library::update_transcription_status()`
- **RAG/Chat** → reads transcripts via Library, updates `chat_status` via Library; stores chunks/embeddings in its own separate database
- **Frontend** → Tauri Commands that delegate to Library internally

### Internal Components

1. **API Layer** — public functions for CRUD operations, artifact management, status updates
2. **SQLite DB** — `meetings` table as single source of truth for metadata
3. **Filesystem Manager** — manages per-meeting directories and artifact files

## Data Model

### SQLite Schema

```sql
CREATE TABLE meetings (
    id TEXT PRIMARY KEY,
    title TEXT NOT NULL,
    created_at TEXT NOT NULL,        -- ISO 8601
    duration REAL NOT NULL,          -- seconds
    has_video BOOLEAN NOT NULL,
    file_size INTEGER NOT NULL,      -- bytes
    dir_path TEXT NOT NULL,          -- absolute path to meeting directory
    media_status TEXT NOT NULL,      -- 'present' | 'deleted'
    transcription_status TEXT NOT NULL, -- 'pending' | 'processing' | 'done' | 'failed'
    chat_status TEXT NOT NULL        -- 'not_indexed' | 'indexing' | 'ready' | 'failed'
);
```

### Filesystem Structure

Each meeting gets its own directory:

```
~/Hlusra/recordings/          (configurable base path)
  2026-03-14_a3f2/
    recording.mkv             -- audio+video or audio-only (always MKV)
    thumbnail.jpg             -- auto-generated for video meetings
    transcript.json           -- segments + word-level timestamps
    transcript.txt            -- plain text
  2026-03-14_b7c1/
    recording.mkv
    thumbnail.jpg
    ...
```

## API

### Meeting Lifecycle

The Library owns meeting IDs and directory creation. The recording flow is:

1. Frontend calls `start_recording` → Recorder calls `Library::prepare_meeting()`
2. Library generates UUID, creates meeting directory, returns `PreparedMeeting { id, dir_path }`
3. Recorder uses `dir_path` as output location for GStreamer `filesink` (writes `recording.mkv` directly into Library-managed directory)
4. When recording stops, Recorder calls `Library::finalize_meeting(id, recording_info)` to populate metadata in SQLite

```rust
Library::prepare_meeting() -> Result<PreparedMeeting>    // creates dir, generates ID
Library::finalize_meeting(id: &str, info: RecordingInfo) -> Result<Meeting>  // writes metadata to SQLite
Library::get_meeting(id: &str) -> Result<Meeting>
Library::list_meetings() -> Result<Vec<Meeting>>     // chronological descending
Library::update_meeting(id: &str, update: MeetingUpdate) -> Result<()>
Library::delete_meeting(id: &str, mode: DeleteMode) -> Result<()>
```

```rust
PreparedMeeting {
    id: String,          // UUID generated by Library
    dir_path: PathBuf,   // e.g. ~/Hlusra/recordings/2026-03-14_a3f2/
}
```

### DeleteMode

```rust
enum DeleteMode {
    Everything,    // entire directory + SQLite record
    MediaOnly,     // deletes .mkv + thumbnail, keeps text/embeddings + record
}
```

When `MediaOnly` is used, `media_status` is set to `deleted`. The meeting remains in the gallery with its transcription and chat still functional.

### Artifacts

```rust
Library::save_artifact(id: &str, kind: ArtifactKind, data: &[u8]) -> Result<PathBuf>
```

```rust
enum ArtifactKind {

```

### Status Updates

```rust
Library::update_transcription_status(id: &str, status: TranscriptionStatus) -> Result<()>
Library::update_chat_status(id: &str, status: ChatStatus) -> Result<()>
```

### Thumbnails

- Auto-generated on `create_meeting` when `has_video == true`
- Uses FFmpeg (`ffmpeg-next`) to extract a frame (first keyframe or ~10s mark)
- Saved as `thumbnail.jpg` in the meeting directory

### MeetingSummary (for gallery)

```rust
MeetingSummary {
    id: String,
    title: String,
    created_at: DateTime,
    duration: Duration,
    has_video: bool,
    file_size: u64,
    media_status: MediaStatus,
    transcription_status: TranscriptionStatus,
    chat_status: ChatStatus,
}
```

### MeetingDetail (for meeting page)

```rust
MeetingDetail {
    // everything from MeetingSummary +
    dir_path: PathBuf,
    tracks: Vec<TrackInfo>,      // defined in Library types (mic, system)
    transcript: Option<String>,  // plain text
}
```

### TrackInfo (owned by Library)

```rust
TrackInfo {
    index: usize,        // 0-based track index in MKV
    label: String,       // "mic" or "system"
    codec: String,       // e.g. "opus"
}
```

## Gallery

- Chronological list (newest first), no search or filters in MVP
- Each card shows: title, date, duration, video/audio indicator, file size, transcription status, chat status
- Video meetings show auto-generated thumbnail
- Audio-only meetings show a generic icon

## Decisions Log

| Decision | Choice | Rationale |
|----------|--------|-----------|
| SQLite crate | rusqlite | Required for sqlite-vec extension compatibility |
| Meeting ID ownership | Library generates UUID | Single source of truth, Recorder receives ID via prepare_meeting |
| File ingestion | Recorder writes directly to Library-managed path | No file move/copy needed, Library creates dir first |
| Metadata storage | SQLite | Standard for desktop apps, good query performance |
| Filesystem layout | One directory per meeting | Isolated, easy to move/delete/backup |
| Access pattern | Library as single gateway | Prevents concurrency bugs, maintains invariants |
| Gallery listing | Chronological, no search | MVP simplicity |
| Deletion | Flexible (everything or media-only) | User can free disk space while keeping text |
| Thumbnails | Auto-generated from video | Better gallery UX with visual previews |
